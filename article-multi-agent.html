🕸️ How Anthropic Built a Multi-Agent Research System — as pictures

This article explains why and how Anthropic built a lead-agent + subagents research system — the exact "Research multi-agent system" pattern tested on CCA-F. Read the pictures, then say each one out loud.

1 · Why use MULTIPLE agents for research instead of one?

Answer: separate agents can explore in parallel, each with its own context window — and it beat a single agent by 90.2%.

Plain words: one agent has to explore ideas one after another. Give each subagent its own separate context window and let them search at the same time, and you cover far more ground, far faster.

💡 Remember it as: many small notebooks beat one crowded notebook.

2 · What is the orchestrator-worker pattern?

Answer: a lead agent plans and delegates; subagents do the actual searching and report back.

Plain words: the lead agent never searches the web itself. It breaks the question into pieces, hands each piece to a subagent, then reads what comes back and stitches the findings into one answer.

💡 Remember it as: the lead agent is the editor, not the reporter.

3 · How does the lead agent hand off work well?

Answer: every subagent gets an objective, an output format, tool guidance, and clear boundaries — vague tasks cause duplicated work.

Plain words: a fuzzy instruction lets two subagents wander into the same search. Spelling out the goal, the format, the tools, and where NOT to go keeps every subagent on its own patch of ground.

💡 Remember it as: a good delegation is a mini-contract, not a hint.

4 · Should every query get the same number of subagents?

Answer: no — scale effort to complexity. Simple facts get 1 agent; complex research gets 10+.

Plain words: spinning up ten subagents for a one-fact question wastes tokens. The system embeds simple rules-of-thumb so effort scales up only when the question actually has many independent parts.

💡 Remember it as: match the team size to the job size.

5 · How did they figure out delegation was broken?

Answer: they watched agents think, step by step, in the Console — that revealed failure modes like over-searching or quitting too soon.

Plain words: you can't fix what you can't see. Watching an agent's actual thinking and tool calls, turn by turn, showed exactly where it went wrong — so the fix targeted the real problem, not a guess.

💡 Remember it as: debug the agent like you'd debug a person — watch it work.

6 · Can the AGENT improve its own tools?

Answer: yes — a tool-testing agent rewrote tool descriptions, cutting later agents' task time by 40%.

Plain words: a model that struggles with a badly-worded tool can also explain WHY it struggled. Anthropic let an agent test the tool and rewrite its own instructions — future agents never hit that same wall.

💡 Remember it as: let the model write its own user manual.

7 · Why is multi-agent so much more expensive?

Answer: multi-agent uses about 15× the tokens of a plain chat — only worth it for high-value tasks.

Plain words: every subagent reads, thinks, and writes its own tokens — the costs add up fast. Fifteen times the tokens is fine for a decision worth thousands of dollars, but wasteful for a quick lookup.

💡 Remember it as: multi-agent is a sports car — great for the big race, overkill for a grocery run.

8 · How do you SCORE whether a research answer is good?

Answer: an LLM judge grades against a rubric — accuracy, citations, completeness, source quality, tool efficiency — on small samples first.

Plain words: testing starts small (20 cases is enough to see big swings). An LLM grades outputs against a checklist for speed and consistency, but a human still reviews — the judge can share the same blind spots as the model.

💡 Remember it as: small samples first, a rubric-judge for scale, a human for the blind spots.

9 · Why do small bugs become BIG problems in agents that run a long time?

Answer: agents are stateful — one small failure early on compounds across many later steps.

Plain words: a stateless script that fails just fails once. An agent carries its state forward — a tiny early mistake shapes every decision after it, snowballing. So save checkpoints and resume, don't just retry from zero.

💡 Remember it as: a snowball starts small.

10 · How do you deploy a NEW agent version without breaking agents mid-task?

Answer: rainbow deployments — shift traffic gradually between old and new versions.

Plain words: an agent might be mid-research for minutes. Flip the whole system to a new version instantly and you can break agents that are already working. Rainbow deploys let both versions run side-by-side while traffic slowly moves over.

💡 Remember it as: don't swap the runway while a plane is landing.
