📝 A real learner's definition, worth keeping: "A framework is a reusable set of code and rules that makes common tasks easier to do in a standard way." The catch: its abstraction layers hide prompts, tool calls and errors — so you must understand them to debug. And the mental model that unlocked the patterns for you: the augmented LLM is the engine; chaining, routing, parallelization, orchestrator-workers and evaluator-optimizer are different circuit designs wired around it.

1 · Workflow or agent — what's the real difference?

Answer: who holds the map. Workflows follow YOUR predefined code path; agents draw their own map as they go.

Plain words: in a workflow, you already decided every step ahead of time — the model just fills in the blanks. In an agent, the model itself decides what to do next, turn by turn, based on what it sees.

💡 Remember it as: workflow = you hold the map. Agent = it holds the map.

2 · When should you NOT reach for an agent?

Answer: when a single optimized LLM call (+ retrieval + examples) is already enough — agents cost more and take longer.

Plain words: agents cost more money and more time than a single call. Before reaching for one, ask: could one well-crafted prompt with a bit of retrieval and a few examples already solve this? If yes — stop there.

💡 Remember it as: find the simplest solution first; agents are a LAST resort, not a first instinct.

3 · Should you use a framework (LangGraph, etc.)?

Answer: frameworks are a glass box vs magic box trade — easy to start, but they can hide the prompts and errors underneath. Start raw; add a framework only once you understand what it's hiding.

Quick recap: frameworks like the Claude Agent SDK, Strands (AWS), Rivet, and Vellum make simple agents easy to build — but the abstraction layer can obscure the underlying prompts, tool calls, and errors, which makes debugging harder when things go wrong. Many agent patterns are only a few lines of raw API code — Anthropic's own advice is to start there, and if you do use a framework, read enough of its source to know what it's really doing underneath.

💡 Remember it as: glass box first, magic box later — never trust a magic box you haven't opened.

4 · What is "the augmented LLM"?

Answer: a plain LLM plus retrieval + tools + memory — the basic building block every workflow and agent is made of.

Plain words: on its own, an LLM only knows what's in its training and what you typed. Give it retrieval (look things up), tools (do things), and memory (remember things) and it becomes the building block every pattern in this article is made from.

💡 Remember it as: LLM + retrieval + tools + memory = the augmented LLM, the Lego brick of every pattern below.

5 · Workflow pattern: Prompt Chaining

Answer: a sequence of calls, each one processing the previous step's output — with an optional check ("gate") in between.

Plain words: instead of one giant prompt trying to do everything, you run small prompts in a row — each one focused on a single step. You can even insert a code check ("gate") between steps to catch problems early.

💡 Remember it as: a relay race — the baton (output) passes step to step, with an optional referee (gate) checking the handoff.

6 · Workflow pattern: Routing

Answer: classify the input first, then send it down a specialized path built for that type.

Plain words: not all questions need the same handling. A router looks at the input first and sends it to the right specialist — a cheap fast model for easy stuff, a stronger model for hard stuff, a dedicated path for refunds.

💡 Remember it as: a receptionist — one look, then the right door.

· deep-dive: Prompt techniques →

7 · Workflow pattern: Parallelization

Answer: run things at the same time, two flavors — sectioning (split the task) and voting (repeat the task).

Plain words: sectioning splits ONE task into independent pieces done at once (e.g. write the reply while a separate call screens it for policy issues). Voting runs the SAME task several times and compares the results for confidence. The aggregator is the plain-code step that merges the parallel answers into ONE result — e.g. "flag it if at least 2 of 3 reviewers say unsafe."

💡 Remember it as: sectioning = divide the work; voting = repeat the work and compare notes — and the aggregator is the vote-counter.

8 · Workflow pattern: Orchestrator-Workers

Answer: a central LLM dynamically breaks down the task and delegates — subtasks are NOT predefined like in parallelization.

Plain words: in parallelization, YOU decided the subtasks ahead of time. Here, the orchestrator LOOKS at each input and decides on the spot how to split it up — good for coding across many files or research that needs different sources each time.

💡 Remember it as: a foreman who plans the job site fresh each morning, not from a fixed blueprint.

9 · Workflow pattern: Evaluator-Optimizer

Answer: one LLM generates, another LLM critiques — loop until the critic says it's good.

Plain words: like a writer and an editor going back and forth. The generator produces a draft, the evaluator judges it against clear criteria, and if it's not good enough the feedback goes back for another round — until it passes.

💡 Remember it as: writer + editor on repeat, until the editor says "ship it."

10 · What does a full autonomous agent loop look like?

Answer: plan → act with tools → read ground truth → repeat — with checkpoints for a human and a max-iteration backstop.

Plain words: a true agent keeps looping on its own — plan, act, check what actually happened in the real world, plan again — until the job is done. Because step count is unpredictable, you need human checkpoints, stopping conditions, and a safe sandbox since mistakes can compound.

💡 Remember it as: plan, act, look, repeat — with a leash (checkpoints) and a fence (sandbox).

11 · What are the three core principles for building agents?

Answer: keep it simple, keep it transparent, and design the agent-computer interface with real care.

Plain words: the best agents aren't the most clever — they're the ones kept as simple as the task allows, that show their work so a human can follow along, and where the tools themselves are designed as carefully as the prompts.

💡 Remember it as: simple, see-through, and speak clearly to your tools.

12 · Appendix: how do you "prompt-engineer" your tools?

Answer: write tool docs like a docstring for a junior dev, test them for real, and make mistakes impossible (poka-yoke) — like requiring absolute file paths.

Plain words: tool descriptions are prompts too — write them as clearly as you'd explain a task to a new hire, and test them. Best of all: redesign the tool so the mistake literally can't happen, like Anthropic's real fix of forcing absolute file paths instead of relative ones.

💡 Remember it as: poka-yoke your tools — don't warn against mistakes, make them impossible.
