1 · Why is a tool different from a normal function?

Answer: a tool is a contract with a non-deterministic caller — the agent can misread it, misuse it, or pick the wrong one.

Plain words: when code calls code, the call is exact. When an agent calls a tool, it's guessing based on a description — it might grab the wrong tool or fill in odd values. You must design for that guessing.

💡 Remember it as: a tool is a contract with a guesser, not an order to a robot.

2 · How do you prototype a tool before trusting it?

Answer: build it fast, wrap it in a local MCP server, then watch a real agent use it and note the rough edges.

Plain words: don't design a tool in your head. Wire up a quick version, hand it to a real agent, and watch where it stumbles. Those stumbles tell you exactly what to fix.

💡 Remember it as: prototype fast, then watch it trip.

3 · What makes a GOOD evaluation task for a tool?

Answer: realistic data + many tool calls + a verifiable answer — not a toy sandbox.

Plain words: a single easy test proves nothing. A good eval forces the agent to chain several real tool calls on real-ish data, and you can check the final answer against a known-correct result.

💡 Remember it as: make the eval sweat — dozens of calls, real data, a real checkable answer.

4 · Why consolidate several API calls into ONE tool?

Answer: bundling related steps into one tool saves context and gets the job done in fewer turns.

Plain words: exposing every raw endpoint makes the agent do the planning AND the plumbing. Bundle multi-step work — find + book, or gather-everything-about-X — into a single higher-level tool.

💡 Remember it as: few thoughtful tools beat many thin API wrappers.

5 · What is namespacing, and why does it matter?

Answer: grouping tools under a shared prefix so the agent can tell similar tools apart.

Plain words: when you have many tools that sound alike, group them by a shared prefix — by service (asana_, jira_) or by resource (asana_projects_, asana_users_). This helps the agent pick the right one fast.

💡 Remember it as: namespacing is folders for tools — like naming files so they sort together.

6 · What should a tool RETURN — raw data or something else?

Answer: return high-signal, meaningful context — not a raw technical dump.

Plain words: agents read text, not database schemas. Prefer name over an opaque id, image_url over 256px_image_url, file_type over mime_type — plain words the model already understands.

💡 Remember it as: return what a person would say, not what a database would store.

7 · How do you keep tool responses token-efficient?

Answer: paginate, filter, truncate with sensible defaults — and let the agent choose response detail.

Plain words: big raw dumps waste tokens and hide the answer. Give tools sensible default limits, filters, and truncation — and when you truncate, say so and hint how to get more.

💡 Remember it as: don't hand over the whole filing cabinet — hand over the one folder, with a note if you trimmed it.

8 · Why do small wording changes to a tool description matter so much?

Answer: the description IS prompt engineering — it's loaded straight into the agent's context every time.

Plain words: the model never sees your code — only the description. Write it like onboarding a new hire: spell out formats, define jargon, name parameters unambiguously (user_id, not user).

💡 Remember it as: the description is the whole interview — the agent never reads your source code.
