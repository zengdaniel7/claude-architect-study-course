1 · What turns a chat model into an agent?

Answer: tools + a LOOP. (JSON output is just data — it doesn't make anything an agent.)

Plain words: a chat model answers once and stops. Give it tools (things it can do) and a loop (act → look at the result → act again), and now it can keep working until the job is done. That's an agent.

💡 Remember it as: model + tools + loop = agent. JSON is just paper — paper doesn't do things.

2 · Why are nullable fields useful in extraction?

Answer: they let the model say "not present" instead of making something up.

Plain words: if a field must be filled, the model will fill it — even by inventing. Making the field nullable (allowed to be empty) gives the model a truthful way out.

💡 Remember it as: null is an escape hatch for honesty. No hatch → the model invents.

3 · A required fact is missing from the source — what should happen?

Answer: mark it missing or send it to a human. Never invent it.

Plain words: "I couldn't find it" is useful information — a human can go look. A confident made-up value is poison: it looks real and nobody checks it.

💡 Remember it as: an honest blank beats a confident lie.

· deep-dive: Reliability thread →

4 · What is prompt chaining?

Answer: breaking a task into sequential steps — each step uses the previous step's output.

Plain words: instead of one giant prompt doing everything (and messing some of it up), you run small prompts in a row. Each one does one clear job and hands its result to the next.

💡 Remember it as: a relay race — the baton is the output.

· deep-dive: Prompt techniques →

5 · A tool times out — what's the BEST error info to send back?

Answer: is_error: true + a message that tells the model what to try instead. (⚠️ "isRetryable" is NOT a real API field — classic exam trap.)

Plain words: the model can only fix what it can read. A good error says what failed and hints what to do next — so the next loop turn goes better.

💡 Remember it as: an error message is an instruction for the NEXT try.

6 · What are path-specific rules for?

Answer: loading different conventions for different parts of a project — automatically, by file path.

Plain words: a project isn't one neighborhood. UI code, API code and tests each have their own house rules. Path-specific rules attach the right rulebook to the right folder — automatically.

💡 Remember it as: different neighborhoods, different rules.

7 · Why are fixed iteration caps a weak PRIMARY stop method?

Answer: a cap doesn't understand the task — it can stop too early or too late. Check stop_reason instead; keep the cap only as a backstop.

Plain words: the counter can't see whether the work is done. stop_reason is the model literally telling you its state. A cap is still fine as a safety backstop — just never as the main brake.

💡 Remember it as: the cap is a smoke alarm, not the off switch.

· deep-dive: The Agent Loop →

8 · When do you enforce a rule in CODE, not just the prompt?

Answer: when the rule MUST hold every time (safety-critical) — use hooks / code gates.

Plain words: a prompt is advice — the model follows it almost always. Code is physics — it happens every time. Money, safety, permissions → the rule goes in a hook, and blocked actions go to a person.

💡 Remember it as: prompts persuade; hooks enforce.

· deep-dive: Support-agent system →

9 · What's the risk of long chats full of verbose tool results?

Answer: important info gets buried in the middle and overlooked — "lost-in-the-middle."

Plain words: models pay most attention to the start and end of the context. A fact drowned in the middle of 40-field tool dumps effectively disappears — so keep context tight and restate the facts that matter.

💡 Remember it as: the middle is where facts go to hide.

10 · When is batch processing a good fit?

Answer: when the work can tolerate delayed results — nobody is waiting on the answer.

Plain words: a customer in a chat can't wait until 3 AM — that's real-time. A pile of documents due tomorrow doesn't care when it's processed — that's batch, at half the cost.

💡 Remember it as: if nobody's waiting, batch it and save half.

11 · What's a good multi-pass code-review pattern?

Answer: review file-by-file first (local bugs), then one integration pass across files.

Plain words: a single pass over everything at once misses both kinds of bugs. Look closely at each tree first, then step back and check the forest — how the files talk to each other.

💡 Remember it as: first the trees, then the forest.
